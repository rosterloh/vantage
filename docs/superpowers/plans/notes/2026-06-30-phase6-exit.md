## Fleet stats — session-derived, reconciles on ungraceful drop (Task 1)

`GET /stats` returns `FleetStats { providers_online, consumers_connected }` read off
the existing `Registry` (TTL-pruned) and `Sessions`. The consumer-count reconciliation
and orphan-close on drop were already handled by the coordinator's socket-drop
teardown — verified, no teardown code was added.

```
A) expect providers=1 consumers=2: {"providers_online":1,"consumers_connected":2}
B) after SIGKILL c2, expect consumers=1: {"providers_online":1,"consumers_connected":1}
C) after robot gone (>15s ttl), expect providers=0: {"providers_online":0,"consumers_connected":0}
```

`kill -9` on a client is an ungraceful disconnect (no clean close message); the count
still decrements because the client WS receive loop exits on the socket close and runs
`sessions.close`. The robot's registry entry expires via the 15 s heartbeat TTL.

## Bidirectional control channel — reserved in one SDP, no renegotiation (Task 2)

The robot (offerer) creates a second `control` data channel alongside `telemetry` in
the same negotiation, unreliable/unordered (`ordered=false, max-retransmits=0`,
latest-command-wins). The client routes incoming channels by label and sends
`ControlMsg` via `Peer::send_control`; the Slint UI maps WASD→`Move` + a 100 ms
keepalive, with an env-driven headless stub (`VANTAGE_CONTROL`, `VANTAGE_CONTROL_STOP_MS`).

Headless round-trip (one client, `VANTAGE_CONTROL=move`):

```
client:  status: connected
client:  video frame 640x480 (#1)          # video + telemetry unaffected
robot:   control from sess-…: Move { linear: 0.5, angular: 0.0 }   (x48 over ~5s)
```

Video and telemetry flow on the same connection while control streams up — the second
DC did not force renegotiation (the SCTP m-line already existed; the channel is
established in-band via DCEP).

## Teleop disconnect failsafe — the safety gate (Task 3)

Per-session `Watchdog` (pure, clock-injected, 4 unit tests) armed on connect. A command
is acted on **only** while `is_live`; inbound control feeds the watchdog then acts;
a 100 ms tick trips stale sessions; disconnect disarms + enters safe state. This is the
first point the robot treats a command as live — the gate the §6 wording requires.

**Staleness** (client sends `Move`+keepalive for 2 s, then stops while staying connected):

```
15:132  acting on control sess-…: move       # last command before the client goes quiet
15:683  safe-state entered: sess-… (control stale)   # ~550ms later (500ms timeout + ≤100ms tick)
```

One safe-state line (idempotent), and no `acting on control` after it. Trip latency
~550 ms matches the 500 ms `CONTROL_TIMEOUT` target within tick granularity.

**Disconnect** (`kill -9` mid-command):

```
09:583  acting on control sess-…: move
09:684  acting on control sess-…: move
09:782  acting on control sess-…: move
09:913  safe-state entered: sess-… (disconnect)    # last line — no acting afterward
```

**Recovery** (a momentary stall self-heals when input resumes) is covered by the
`feed_keeps_it_live_and_recovers_after_a_trip` unit test.

## mDNS LAN fast-path + offline direct connect — verified live (Task 4)

Robot advertises `_vantage._tcp` (via `mdns-sd`, pure-Rust — no Avahi) and runs a
direct signalling listener (`PeerWs` server), surviving a coordinator-unreachable
start. The direct path reuses the shared `RobotMedia` engine and the **same teleop
watchdog**, so control stays safe offline. The client falls back to an mDNS browse +
direct WebRTC connect when the coordinator WS is refused.

**With NO coordinator running** (both robot and client pointed at a dead coordinator):

```
robot:   direct LAN signalling listener on port 35077
robot:   advertised _vantage._tcp.local. instance robot-1 on port 35077
robot:   coordinator unreachable (Connection refused) — serving LAN clients via mDNS/direct only
client:  coordinator unreachable (Connection refused) — trying mDNS LAN fast path
client:  mDNS: discovered robot-1 at 100.88.137.20:35077
client:  status: connected
client:  video frame 640x480 (#1 … #90+)          # streaming with no coordinator
robot:   acting on control direct-0: move linear=0.5 angular=0   # teleop works offline
```

Multicast is available on this host, so the **live browse + direct connect is verified
here, not host-deferred**. Host candidates carry the same-LAN link (STUN-only ICE, no
TURN). The coordinator path is unchanged and regression-checked (stats `1/1`, video
streams normally).
