## Decoder factory (client, hardware decode + software fallback)

Runtime selection over `nvh264dec → nvdec → vah264dec → vaapih264dec → d3d11h264dec
→ vtdec_h264 → avdec_h264`. This host has no hardware decoder, so it selects
`avdec_h264` (logged: `selected H.264 decoder: avdec_h264`). Hardware arms are
written from GStreamer docs and would be exercised on a VAAPI/NVDEC host (the
decode-side twin of Phase 4a's encoder factory).

## Encode-once fan-out to two viewers (headless)

`RobotMedia` capture/encode engine: `source → I420 caps(640x480@30) → tee`, with the
encode chain `queue(leaky) → x264enc (built ONCE) → h264parse → rtph264pay → rtpcaps →
rtptee`. Each client session adds one `queue → webrtcbin` tapped off `rtptee`.
Ran coordinator + robot + two headless clients (client 2 staggered ~3 s after client 1):

```
client1 frames: 15            # decoded via WebRTC
client2 frames: 12            # decoded via WebRTC
encoders selected (MUST be 1): 1
encoder line: selected H.264 encoder: x264enc
```

The encoder is built/selected **exactly once** while **two** clients receive video —
"Encode once, fan out to multiple consumers" / "encoder count does not increase".

## Immediate startup for new viewers (keyframe-on-join)

On each consumer join an upstream `UpstreamForceKeyUnitEvent` is sent to the shared
encoder (the "force-key-unit not handled" warning did **not** fire ⇒ it was honoured).
Client 2's timeline:

```
21:00:15.937560  status: connecting to Atlas
21:00:16.100786  status: connected            # WebRTC/DC established
21:00:16.326001  video frame 640x480 (#1)     # first decoded frame
```

≈ **225 ms** from connection-established to first decoded frame — well under the
~1 s periodic-IDR backstop (`key-int-max=30` @30fps). New viewers render promptly
rather than waiting for the next periodic keyframe.

## Demand-driven teardown — survivor unaffected

Killing client 2 mid-stream:

```
client disconnected: sess-18bd5a64abfb3059
removing consumer sess-18bd5a64abfb3059        # robot_media blocking pad-probe teardown
client1 frames before c2-kill: 15
client1 frames after  c2-kill: 22              # client 1 keeps receiving
client1 survived: YES
robot alive: YES   coordinator alive: YES   errors/segv: 0
```

Only the departing consumer's branch is torn down (IDLE blocking pad probe →
unlink rtptee→queue, remove queue+webrtcbin, release the request pad); the engine
and the surviving viewer are undisturbed, no crash.

## Adaptive bitrate (transport-cc) — wired, scenario host-deferred

`transport-cc` is advertised on the payloader (`transport-cc RTP header extension
added to payloader (ext-id=3)`, via the safe `gstreamer-rtp` `RTPHeaderExtension::set_id`).
`rtpgccbwe` is attached via `webrtcbin`'s `request-aux-sender` **when present**;
this host does not have `rtpgccbwe`, so the guard runs:

```
WARN rtpgccbwe absent — static bitrate (adaptive bitrate host-deferred)
```

Streaming continues at the static bitrate, no engine disruption. The
**bandwidth-falls → bitrate-drops** scenario is **host-deferred** — verify on a host
with `rtpgccbwe` installed (ships in `gst-plugins-rs` net/webrtc). It is NOT claimed
as passed here.

> Latent (tracked): the adaptive bitrate setter clamps in kbit/s (correct for
> `x264enc` and most encoders). On a Jetson `nvv4l2h264enc` (bits/s units) the
> divide-by-1000 would mis-set the bitrate — only reachable on Jetson *with*
> `rtpgccbwe`, so unreached on this host. Add an encoder-name branch if Jetson +
> adaptive bitrate becomes a target.

## Camera / raw coexistence with two viewers

Verified via the default test source (the Phase 4a/4b convention; no change to the
frame-production or raw-branch path in Phase 5). The raw pre-encode branch is
unchanged from 4a/4b and is independent of the per-consumer `webrtcbin` branches, so
"Camera not monopolised" continues to hold with N>1 consumers. Live camera + ROS raw
on hardware with two viewers remains a carry-over hardware check (see Status).
